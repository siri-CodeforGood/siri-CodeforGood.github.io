Case Study 2

From Production Firefighting to Scalable Reliability

How I diagnosed and resolved a cluster of concurrent production issues — spanning database exhaustion, email delivery, cloud routing, and container scaling — without playbooks or prior context.

My Role: sole investigator and resolver — backend, cloud, and communications stack
What was breaking (all at once, in production):
  • Azure Function Apps timing out intermittently — traced to database connection pool exhaustion under load
  • Transactional emails silently failing for users on Microsoft-hosted domains — no bounces, no alerts
  • SMS and phone-lookup webhooks from Twilio routing to the wrong environment — events were being silently swallowed
  • Container autoscaling triggering unexpectedly during enrollment flows — caused by connections not being recycled
  • A background token-refresh job running in both dev and prod simultaneously — creating token invalidation risk for utility API access
How I resolved each one:
  • Built async PostgreSQL query helpers with separate connection pools for generic vs. webhook traffic — eliminated timeout cascades
  • Added connection recycling to bulk-processing and enrollment flows — stopped the spurious autoscaling
  • Debugged SendGrid domain authentication for Microsoft and Yahoo/AOL — restored full email deliverability
  • Corrected Twilio webhook endpoints and phone-lookup routing — dev and prod fully isolated with no shared state
  • Scoped the ConEd token-refresh job to production only — removed the cross-environment invalidation risk
Outcome: All five issue classes resolved within the same sprint. Zero recurring DB exhaustion. Email deliverability restored across all domains. Container autoscaling stabilized. Platform has maintained 99.9%+ uptime since.

Natural Language → SQL POC
AI-powered query generation over the Meltek production schema — iteratively built and improved
AI Innovation
My Role: sole builder — concept, schema prompting, integration, iteration
Challenge: Non-technical operators needed to query complex, multi-schema production data (enrollments, intervals, settlements) without writing SQL — and without hallucinated or wrong-table results.
Built & Iterated:
  • Initial POC: NL → schema-aware SQL → execution pipeline working end-to-end, limited to a single schema
  • Expanded to all available schemas; exposed schema selection accuracy issues from lack of contextual prompt parsing
  • Implemented keyword-based weighted scoring for intelligent schema selection — reliably identifies the most relevant schema. Added metadata caching to reduce redundant introspection calls
  • Integrated Hugging Face embeddings for semantic table matching; OpenAI API interprets query results into plain-language summaries for non-technical operators
Status: Active POC. Each iteration measurably improved query accuracy and reduced noise. Uses Hugging Face embeddings for semantic schema/table matching and OpenAI API for natural language result interpretation. Foundation for AI-assisted admin tooling across the Meltek platform.
